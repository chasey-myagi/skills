# TDD — 红绿重构，垂直切片

TDD 不是"测试覆盖率"，是一种**设计手段**：先用一个测试说清楚"系统该做什么"，再写最少的代码让它成立。每个测试是一颗 tracer bullet（曳光弹），打通一条端到端的路。

这是 skills repo 开发质量链的**入口**，写完测试往下游接质量门：

```
tdd（写测试 + 实现）
  → test-review    审测试质量（覆盖、边界、有没有在测实现细节）
  → code-review    审实现质量（正确性、安全、架构）
  → linus-review   审工程品味（能不能消除特殊情况）
```

## 一条铁律：测行为，不测实现

测试验证的是**通过公共接口能观察到的行为**，不是内部怎么实现的。代码可以整个重写，测试不该动。

```js
// ✅ 好：测的是调用方在意的行为，走公共接口
test("用户能用合法购物车结账", async () => {
  const cart = createCart();
  cart.add(product);
  const result = await checkout(cart, payment);
  assert.equal(result.status, "confirmed");
});

// ❌ 坏：测的是内部怎么实现的，换个写法就挂
test("checkout 调用了 paymentService.process", async () => {
  const spy = mock(paymentService);
  await checkout(cart, payment);
  assert.equal(spy.process.callCount, 1);   // 重构没改行为，它也会红
});
```

**警报信号**：你重构了内部、行为没变，测试却挂了——那它测的是实现，不是行为，是负债。

坏味道清单：mock 内部协作者、测私有方法、断言调用次数/顺序、绕过接口直接查数据库来验证。

## 反模式：水平切片（最常见的 TDD 错误）

**不要先写完所有测试，再写所有实现。** 这是"水平切片"，它产出垃圾测试：

- 批量写的测试测的是**想象的**行为，不是**真实的**行为；
- 你会测到"形状"（数据结构、函数签名）而不是用户可见的行为；
- 测试对真实变化变得不敏感——行为坏了它还绿，行为没事它反而红。

**正确做法：垂直切片 / tracer bullet。** 一个测试 → 一个实现 → 再来。每个测试都回应你从上一轮学到的东西——因为代码刚写完，你确切知道什么行为重要、怎么验证。

```
错（水平）:  RED: test1..test5     GREEN: impl1..impl5
对（垂直）:  test1→impl1  test2→impl2  test3→impl3  ...
```

完整真实演示见 [examples/tdd-session.md](examples/tdd-session.md)——`parseDuration` 从 0 到 5 个测试的红绿重构全过程，每步真实测试输出都在，可复现。

## 工作流

### 1. 规划（动手前）

- [ ] 和用户确认要改哪些**公共接口**；
- [ ] 和用户确认**测哪些行为、排优先级**——你测不完所有东西，聚焦关键路径和复杂逻辑，不是每个边界都测；
- [ ] 把要测的**行为**列出来（不是实现步骤）；
- [ ] 接口设计成好测的样子：**接收依赖、别自己造**（依赖注入），**返回结果、别制造副作用**，接口面尽量小。

问用户："公共接口长什么样？哪些行为最该测？"

### 2. Tracer Bullet（第一颗曳光弹）

写**一个**测试，确认系统的**一件事**：

```
RED:   写第一个行为的测试 → 失败
GREEN: 写最少的代码让它过 → 通过
```

证明这条路端到端打通了。

### 3. 增量循环

对每个剩下的行为，重复：

```
RED:   写下一个测试 → 失败
GREEN: 刚好够过的代码 → 通过
```

规则：一次一个测试；只写够过当前测试的代码；不要预判未来的测试；测试盯住可观察的行为。

### 4. 重构（全绿之后）

所有测试通过后再找重构机会：消除重复、把复杂度藏到简单接口后面（deep module = 小接口 + 深实现）、把逻辑移到数据所在处、给基本类型套值对象。

**永远不要在红灯时重构。先到绿灯。** 每步重构后立刻重跑测试——绿了才算重构成功，红了就回退。

## mock 只在边界

只 mock **系统边界**：外部 API（支付、邮件）、数据库（能用测试库更好）、时间/随机、文件系统。

**不要 mock**：你自己的类/模块、内部协作者、任何你能控制的东西。mock 内部 = 把测试焊死在实现上，重构必挂。

边界处用依赖注入 + SDK 风格接口（每个外部操作一个具体函数），让边界天然好 mock，也省掉测试里的条件分支。

## 每轮 checklist

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只走公共接口
[ ] 测试能扛住内部重构
[ ] 代码是过这个测试的最少量
[ ] 没有加投机的、没人要的功能
```

## 反例黑名单

- 不要水平切片（先写完所有测试再写实现）。
- 不要 mock 自己的代码。
- 不要测私有方法 / 内部结构 / 调用次数。
- 不要在红灯时重构。
- 不要为了凑覆盖率写"测形状"的空测试。
- 不要一个测试断言一堆不相关的事——一个测试一个行为。

## 接下来

写完测试和实现，把它们交给下游质量门：`/test-review` 审测试够不够、`/code-review` 审实现稳不稳、`/linus-review` 看有没有能消除的特殊情况。TDD 保证你写了测试，质量门保证测试和实现都够好。
